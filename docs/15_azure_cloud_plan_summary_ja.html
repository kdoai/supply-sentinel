1. 何を実現するか

審査で伝える中心メッセージは、予測精度ではなく「初動時間の短縮」と「外部リスクの自社影響への翻訳」。

外部リスクを検知

ニュース、サプライヤ通知、物流、災害、価格情報から供給不安の兆候を拾う。

ニュースPDF/メール物流

自社影響に翻訳

BOM、在庫、代替材、受注を照合し、どの製品・工場・顧客に何日後に影響するかを出す。

最短5日影響額7.8百万ドル/月

初動を準備

AIは勝手に発注や顧客通知をしない。担当タスク、承認事項、管理職レポートを準備する。

Human-in-the-loop管理職レポート

一言でいうと、Supply Sentinel は「世の中の供給不安」を「自社の製品Aが5日後に止まるリスク」へ変換する。

Scheduler Agent定期実行、run_id発行、前回状態読み込み

Impact Assessment AgentBOM・在庫・受注・代替材を照合し、自社影響を算出
ルール中心

GPT利用

7. コスト方針

Hackathonでは最小能力でよい。スケールできる設計だけ残し、初期は従量課金とキャッシュで抑える。

Functions

Consumption/Flex Consumption。Always Readyは使わない。デモは手動または1時間ごと。

GPT

GPT-4o mini相当を優先。候補イベントだけ抽出し、高リスク時だけレポート生成。

Cosmos / Storage

Cosmos DB Free TierまたはServerless。Blobは小容量、古い成果物はTTLや削除で管理。

Azure AI Search、AKS、VM、常時起動Container Appsは初期構成から外す。月1万円以内を狙うなら固定費を増やさない。

8. GitHub CI/CD

既にworkflow雛形を追加済み。Azure接続情報を入れればデプロイを手動実行できる。

CI

  • push / pull_requestで実行
  • Node 20 セットアップ
  • 構文チェック
  • ユニットテスト
  • Web assets build

Azure Deploy

  • workflow_dispatchで手動実行
  • GitHub OIDCでAzure login
  • Azure Functionsへデプロイ
  • RUN_MODEをdemo/cloudで切替
  • 長期シークレットを置かない

9. 実装ロードマップ

Hackathonまでの優先順位。まず「Azureで定期実行され、AIで構造化し、自社影響に変換する」ことを見せる。

Step 1
Azure Function Appを作成

Timer TriggerとHTTP Triggerの最小実行を通す。

Step 2
Blob Storage入力へ切替

data/samplesをBlob inputに置き、同じスキーマで読み込む。

Step 3
Azure OpenAI接続

Risk Extraction AgentとResponse Planner AgentだけGPTを使う。

Step 4
Cosmos DB状態管理

runs、risk_events、impact_assessments、alert_stateを保存する。

Step 5
DashboardをAPI接続

/api/latestまたはBlob JSONから最新結果を取得する。

Step 6
デモ台本の固定

ナフサで見せ、最後に材料データ差し替えで横展開可能と説明する。
