# 5分デモ台本

## デモの目的

Supply Sentinel が、外部の供給リスクを「ニュース要約」ではなく「自社のどの製品・顧客・工場が、いつ止まるか」という業務影響へ翻訳し、人が承認する初動を高速に準備する流れを5分で見せる。

ナフサ供給リスク1シナリオを、3画面UI（監視ダッシュボード / 影響分析 / 初動対応）で気持ちよく通す。機能の幅ではなく、判断の速さで勝つ。

## 画面とデモ操作の前提

- 画面は左サイドバーの3つ: 「監視ダッシュボード」「影響分析」「初動対応」。
- 監視ダッシュボードには「エージェント巡回デモ」操作（巡回デモ開始 / リセット）と監視対象スイッチャー（ナフサ / 包装フィルム / 半導体接着材）がある。
- 巡回デモはスコアが 34 → 58 → 74 → 82 → 82 の5段で進み、地図・調達構成・業務影響KPI・監視イベントが順に更新される。
- 地図のルートをクリックすると、調達構成の該当行と業務影響KPIが連動ハイライトする。
- 起動URL例: `?demo=play` で自動再生、`?view=response` で初動対応から開始、`?material=naphtha` で監視対象を固定。

## 0:00 - 0:30 課題提示

話すこと:

> 原材料の供給リスクは、ニュースが出た瞬間に問題になるのではありません。「これは自社のどの製品・顧客・工場に、いつ影響するのか」が分からない時間こそが、業務を止めるリスクです。Supply Sentinel は、その空白の時間を埋める早期警戒エージェントです。

見せる画面 / 操作:

- 「監視ダッシュボード」を開いた状態（巡回デモは待機 = baseline、スコア34）。
- 世界地図に供給ルート、サイドに「リスク判定」「業務影響」、下部に「調達構成」「監視イベント」が並ぶ全体像を一望させる。
- 「いまは平常。ここから外部シグナルが入った瞬間を見ます」と前置きする。

## 0:30 - 1:00 定期巡回エージェント

話すこと:

> Supply Sentinel は Azure Container Apps Job として 6時間ごと（cron `0 */6 * * *`）に定期巡回します。常時監視しているように見えますが、実装はシンプルな定期実行です。常駐サーバを持たないので、コストも抑えられます。

見せる画面 / 操作:

- 監視ダッシュボードの「エージェント巡回デモ」で「巡回デモ開始」を押す。
- 「監視イベント」が baseline（06:30 定期巡回を開始）から流れ始める様子を見せる。
- 「巡回が起動し、ニュース・サプライヤ通知・物流/価格シグナル・社内在庫を順に読み込みます」と説明。

## 1:00 - 1:45 AIリスク抽出

話すこと:

> 巡回が拾った生テキスト（業界ニュースとサプライヤ通知）から、AIが構造化リスクイベントを抽出します。対象原材料、リスク種別、深刻度、遅延5-7日、次回割当70%、確度（confidence）まで、決まったスキーマで返します。これは要約ではなく、後段の業務照合に使える機械可読データです。

> 抽出の正直なところをお話しします。いまのデモは信頼性重視で、同じスキーマを決定論的に返す demo mode です。本番では完全に同じスキーマを Azure OpenAI（gpt-5.4）が抽出し、補助的なサブ処理は gpt-5.4-mini を使います。現在は Azure 側の TPM クォータを申請中で、承認され次第そのまま差し替わる設計です。

見せる画面 / 操作:

- 巡回を news（06:31 / スコア58）→ supplier_notice（06:32 / スコア74）へ進める（自動再生 or リセット後に段を進める）。
- 「初動対応」画面に切り替え、「検知根拠」パネルで抽出された根拠リスト（ニュース見出し、5-7日遅延、割当70%、通知件名）を見せる。
- 対比として、抽出されるリスクイベントのスキーマを口頭または資料で示す:

```json
{
  "material": "naphtha",
  "risk_type": "allocation",
  "region": "Asia",
  "delay_days_min": 5,
  "delay_days_max": 7,
  "allocation_rate_percent": 70,
  "severity": "high",
  "confidence": "high"
}
```

## 1:45 - 2:45 自社影響への翻訳

話すこと:

> ここが Supply Sentinel の核です。外部リスクを、自社の BOM・在庫・受注・代替材・調達ルートに照らして、業務影響に翻訳します。今回の影響は、製品3種、顧客3社、工場2拠点。最も切迫しているのは千葉工場で、在庫は残り5日です。

> ナフサの調達は4ルート。そのうち3ルートが影響を受け、影響を受ける調達比率は65%、月間 780万ドル（全体1,200万ドル）に相当します。残る1ルート（サウジアラビア・中東）は影響を受けず、代替の出し手として残ります。

見せる画面 / 操作:

- 巡回を impact_assessment（06:33 / スコア82）まで進める。
- 「影響分析」画面に切り替え、4パネルを順に示す:
  - 「業務フロー」: 原材料 → サプライヤ → 工場 → 製品 → 顧客のつながりで、被影響経路が赤系で強調される。
  - 「影響受注一覧」: SO-1001（自動車部品A社・樹脂A・高）、SO-1002（包装材B社・溶剤B・高）、SO-1003（化学品C社・コーティングC・中）。
  - 「BOM・在庫照合」: 千葉工場 残5日、大阪工場 残10日。
  - 「代替材候補」: NAP-ALT-01（承認済・リードタイム10日・樹脂Aのみ）、NAP-ALT-02（評価中）。
- 監視ダッシュボードに戻り、地図で被影響ルート（R-NAP-02 / R-NAP-03 / R-NAP-04）をクリック → 調達構成の該当行と業務影響KPI（影響調達比率・影響額・ルート数）が連動ハイライトする動きを実演する。
- 「外部の出来事が、自社のこの受注・この工場・この顧客に直結することを、クリックひとつで追えます」と締める。

## 2:45 - 3:30 リスクスコアと初動

話すこと:

> 総合リスクスコアは 82/100、危険度 High です。重要なのは、これを AI の主観で決めていない点です。スコアは透明な業務ルールで積み上げます。外部イベントの深刻度、サプライヤ通知の確度、在庫残日数リスク、顧客・受注の優先度、代替材の有無。だから「なぜ82なのか」を人に説明できます。

> このスコアを根拠に、推奨初動と担当タスクを提示します。誰が、いつまでに、どの状態か。担当・期限・状態まで埋めて、現場がすぐ動ける形にします。

見せる画面 / 操作:

- 監視ダッシュボードの「リスク判定」パネルでスコア 82/100・High を示す。
- 「初動対応」画面に切り替え、次を見せる:
  - 「推奨初動」: サプライヤへ割当数量と出荷予定を確認 / 高優先度の在庫を確保 / 承認済み代替材 NAP-ALT-01 の適用可否を確認 / 高優先度顧客向け一次説明文案を準備 / 5-7日遅延に対する生産計画の露出を評価。
  - 「担当タスク」: 調達（樹脂調達G・本日17:00・進行中）/ 生産管理（千葉工場・24時間以内・着手待ち）/ 品質保証（材料認定・24時間以内・着手待ち）/ 営業（重点顧客担当・承認後・承認待ち）。
- 「スコアの根拠（在庫残日数・顧客優先度・代替材）が、そのまま担当タスクに落ちている」点を強調する。

## 3:30 - 4:20 Human-in-the-loop

話すこと:

> Supply Sentinel は、発注変更やサプライヤ切替を勝手に実行しません。AI は判断材料を揃えて起案するところまで。重要な判断は、必ず人が承認します。

> 承認が必要な判断は4つ。発注内容の変更（承認者: 調達部長）、サプライヤ切替（調達部長 / 品質保証）、顧客への正式通知（営業部長）、生産計画の大幅変更（生産管理責任者）。いずれも承認待ちの状態で並び、承認者が見える形にしてあります。あわせて、管理職向けレポートを自動生成します。

見せる画面 / 操作:

- 巡回を response_ready（06:34 / スコア82）まで進める。
- 「初動対応」画面の「承認が必要な判断」パネルで、4件が承認者付き・承認待ちで並ぶことを見せる（リード文「Supply Sentinel が起案。実行はいずれも人の承認が前提です。」）。
- 同画面の「管理職レポート」パネルで、リスクスコア・影響調達額・影響製品・一次判断（発注変更と顧客通知は人の承認後に実施）が自動でまとまっていることを示す。
- 「AI は速度を、人は責任を持つ。だから現場と経営の両方が安心して使えます」と締める。

## 4:20 - 5:00 締め

締め（docs/07 のクロージングを日本語化）:

> 価値は、未来を完璧に予測することではありません。価値は、どの製品・どの顧客・どの工場が危ないかを早く知り、業務が止まる前に初動を準備できることです。Supply Sentinel は、世の中の供給不安を「自社の製品Aが5日後に止まるリスク」へ翻訳し、人が承認する初動を高速化します。

一言まとめ（One-Liner）:

> Supply Sentinel は、外部の供給リスクを自社の業務影響に翻訳し、初動を加速する早期警戒エージェントです。

横展開の一言（時間が許せば）:

> 監視対象スイッチャーでナフサを包装フィルムや半導体接着材に切り替えても、同じ仕組みで動きます。原材料の追加はコードではなくデータの追加で対応できます。

### Azure構成サマリ（締め前後に1ブロック）

話すこと:

> 構成は月数千円に収める前提の最小構成です。アプリ実行は Azure Container Apps（Consumption）、定期巡回は同じ基盤の Container Apps Job が6時間ごとに実行します。状態は Cosmos DB（Serverless）、コンテナは Azure Container Registry（ACR Basic）、フロントは Storage の静的ウェブサイト。CI/CD は GitHub Actions + OIDC で、長期シークレットは置きません。認証は Entra ID / Managed Identity。月3,000円の Budget アラートでコストを監視します。

構成要素:

- Azure Container Apps（Consumption）: APIとワークフロー実行。常駐させず従量課金。
- Container Apps Job（cron `0 */6 * * *`）: 6時間ごとの定期巡回。
- Azure Cosmos DB（Serverless）: 実行履歴、検知イベント、未解決アラートの状態管理。
- Azure Container Registry（ACR Basic）: コンテナイメージ管理。
- Storage Static Website: ダッシュボードのフロント配信。
- GitHub Actions + OIDC: CI（構文チェック・テスト・Webビルド）と Azure デプロイ。
- Entra ID / Managed Identity: クラウド間認証。アプリ内ログインは作らない。
- Budget アラート（月3,000円）: コスト超過の早期検知。
- AI: 本番は Azure OpenAI gpt-5.4（リスク抽出・レポート生成）、サブ処理 gpt-5.4-mini。現在 TPM クォータ申請中。

### バックアップ手順

- ライブAI呼び出しが不可でも、demo mode（決定論キャッシュ）で同じ画面・同じ数値を再現できる。AI接続が落ちても自動で決定論抽出にフォールバックする設計。
- ネットワークやAzureが使えない場合は、`?demo=play` の静的デモURLで巡回を再生する。
- スコア 34 → 58 → 74 → 82 の5段は demo_events.json に固定されており、毎回同じストーリーで通る。
